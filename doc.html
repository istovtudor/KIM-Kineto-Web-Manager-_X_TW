1. Introducere

1.1 Scopul documentului

Prezentul raport tehnic stabilește specificațiile arhitecturale, funcționale și structurale pentru platforma web de gestiune Gym & Kineto. Structura documentului respectă bunele practici din ingineria software descrise în standardul IEEE SRS (System Requirements Specification), transpus într-un format web semantic bazat pe specificațiile Scholarly HTML.

1.2 Scopul produsului (Product Scope)

Aplicația este o soluție digitală ce optimizează fluxul operațional dintr-un centru sportiv sau o clinică de recuperare. Obiectivul central este corelarea resurselor logistice (săli, aparate), a resurselor umane (antrenori, kinetoterapeuți) și a programărilor efectuate de clienți în funcție de pachetele lor active.

1.3 Definiții și Acronime

  • SRS: System Requirements Specification.
  • REST API: Application Programming Interface bazat pe arhitectura REST, utilizând JSON.
  • GD: Graphics Draw (bibliotecă PHP utilizată pe server pentru generarea de grafice statistice în mod nativ).
  • RBAC: Role-Based Access Control (Controlul accesului bazat pe roluri).

2. Descriere Generală

2.1 Perspectiva Produsului

Sistemul adoptă o arhitectură decuplată. Serverul (Back-end) este scris în PHP nativ și funcționează exclusiv ca un API asincron care procesează datele din baza de date și livrează răspunsuri formatate JSON. Interfața (Front-end) este alcătuită din resurse statice (HTML5, CSS3) animate prin JavaScript asincron (Fetch API).

2.2 Clase de Utilizatori și Matricea de Drepturi

Accesul la acțiuni și date este guvernat strict prin verificarea sesiunii securizate la nivel de server:

Rol Utilizator Funcționalități și Permisiuni în Back-end Directoare Front-end Asociate
Member (Membru) Înregistrare, logare, vizualizare orar, rezervare ședințe pe bază de abonament activ, gestionare profil personal. /auth/, /schedule/, /subscriptions/, profile.html
Trainer (Antrenor) Toate drepturile de membru, configurarea propriei matrici de disponibilitate, vizualizarea agendei de clienți, modificarea sau anularea ședințelor programate de el. /trainer/, /schedule/, index.html
Admin (Administrator) Managementul conturilor (alocare rol de antrenor), configurarea resurselor logistice (săli, echipamente), vizualizarea rapoartelor statistice complexe, export brut de date. /admin/, /reports/, /resources/

3. Arhitectura Sistemului și Structura de Directoare

Pentru a asigura paralelismul muncii în cadrul echipei de doi programatori, proiectul a fost organizat modular, separând complet fișierele de interfață de scripturile care rulează pe server.

3.1 Organizarea Modulelor Front-end

Structura folderului public/ reflectă direct imaginea de organizare a interfeței grafice:

public/ ├── auth/ # Interfețele de Login și Înregistrare (punct de start) ├── admin/ # Panoul de control pentru managementul utilizatorilor ├── schedule/ # Interfața de calendar și rezervare a ședințelor ├── subscriptions/ # Vizualizarea și activarea pachetelor de fitness/kineto ├── trainer/ # Panoul dedicat antrenorilor pentru stabilirea orarului ├── reports/ # Vizualizarea statisticilor dinamice și a fișierelor descărcabile ├── resources/ # Managementul logisticii fizice (săli și aparate) ├── css/ # Stilurile globale aplicate paginilor din aplicație └── js/ # Logica de script pentru apelurile asincrone către API

3.2 Corelarea cu Scripturile Core Back-end

Paginile statice din directoarele menționate mai sus trimit cereri de date către fișierele PHP de procesare aflate la nivelul de bază al serverului:

  • router.php și index.php: Gestionează maparea URL-urilor, setarea MIME-type-urilor corecte (pentru CSS/JS) și redirecționarea utilizatorilor neautentificați.
  • user.php și admin_users.php: Controlează fluxurile de autentificare (login/register) și modificarea drepturilor utilizatorilor.
  • schedule.php și trainer_schedule.php: Gestionează logica orarului și a rezervărilor active.
  • subscription.php: Controlează starea abonamentelor și decrementarea automată a ședințelor.
  • report.php: Motorul analitic care prelucrează statisticile globale.

4. Cerințe Funcționale și Reguli de Business

4.1 Gestiunea Rezervărilor și Evitarea Conflictelor (schedule.php)

Aplicația validează fiecare acțiune pentru a asigura consistența datelor: la salvarea unei ședințe, se apelează metoda hasConflict. Dacă sala sau antrenorul sunt ocupați în acel interval, tranzacția este anulată și se returnează HTTP 409 Conflict. Membrii sunt limitați la o singură rezervare pe zi prin verificarea hasBookingOnSameDay.

4.2 Consumul Atomic al Abonamentelor (subscription.php)

Ședințele sunt mapate pe categorii (fitness, forta, kineto). Când un membru face o rezervare, funcția findSubscriptionForBooking caută un pachet valid. Dacă este găsit, numărul de ședințe rămase este decrementat atomic în baza de date pentru a preveni utilizarea frauduloasă.

4.3 Modulul de Statistică și Export (report.php)

Pentru administratori, serverul compilează datele și folosește biblioteca nativă GD pentru a desena grafice de tip histogramă salvate ca imagini PNG/WebP. De asemenea, modulul permite exportul datelor agregate în fișiere brute de tip CSV sau structuri ierarhice XML.

5. Cerințe Non-Funcționale și Securitate

5.1 Autentificare și RBAC

Parolele sunt securizate exclusiv prin hashing criptografic (password_hash). Sesiunile sunt inițializate în condiții de siguranță (startSecureSession). Fiecare endpoint back-end verifică permisiunile prin funcția de barieră requireRole(), returnând HTTP 403 Forbidden în caz de acces neautorizat.

5.2 Siguranța Datelor și CORS

Toate interogările SQL folosesc tehnica declarațiilor pregătite (PDO) pentru a elimina riscul atacurilor de tip SQL Injection. Pentru a asigura flexibilitatea interconectării componentelor, headerele HTTP includ directive CORS complete (Access-Control-Allow-Origin: *).
